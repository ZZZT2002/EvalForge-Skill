## 快速开始

**环境（一次性）**
```bash
python -m venv zzwork
zzwork/Scripts/activate         # Windows
# source zzwork/bin/activate    # macOS / Linux
pip install -r requirements.txt
cp .env.example .env            # 填 DEEPSEEK_API_KEY（MOCK 模式可不填）
```

**首跑 v1.0 评测集**
```bash
python -m generator.pipeline              # 阶段 2：seeds → candidates_v0.9.json (591 条)
python -m validator.quality_pipeline      # 阶段 3：schema + LLM-judge → candidates_v0.95.json
python -m validator.human_console         # 阶段 4：CLI 抽检 20%（y/n/m/s 决策）
python release.py                         # 发布 benchmark_v1.0.0.json + CHANGELOG
```

**主流水线（v1.0 已存在则幂等跳过）**
```bash
python main.py                  # B 评测 + C 衰退演进 → v1.1 + D 反向注入
```

**只跑评测器单元测试（答辩"技术深度"核心证据）**
```bash
pytest tests/test_scorer.py tests/test_normalizer.py -v   # 三维判分 + 4 种格式容错
pytest tests/ -q                                          # 全套 130 个测试
```

## MOCK vs REAL 模式切换

`config.py` 中 `EVAL_MODE = "MOCK" | "REAL"`：

| 阶段 | MOCK 行为 | REAL 行为 |
|---|---|---|
| 生成器 (D3) | 跳过 LLM 改写，仅做参数膨胀 | DeepSeek 改写 + TF-IDF 去重 |
| 自动质检 (D5) | 跳过 LLM-Judge，仅 schema 强校验 | DeepSeek 三维打分 |
| 评测器 (D8) | `MockAdapter` 按 error_rate 模拟答案 | `DeepSeekAdapter` 真调 function-calling |

> 开发期一律 MOCK 跑通，临展示前再切 REAL（成本 ≤ ¥5）。所有适配器实现共享 `BaseAdapter.call()` 接口，切换无需改判分链路。

## 设计要点速览

- **三维判分**：`0.3·tool_recall(F1) + 0.3·tool_order(LCS) + 0.4·argument_accuracy`（顺序贪心 + `matched_expected_idx` 防刷分）
- **演进策略**（`md5(task_id) % 3` 稳定哈希分配）：
  - `precondition_injection`：头部注入安全前置工具
  - `constraint_tighten`：末尾追加审计通知
  - `adversarial_escalation`：prompt 追加诱导文字 + 强制审计日志
- **反向注入命名空间**：`T_<SKILL>_FB_NNN`（避开原 `_V00..V19` 变体后缀）
- **版本冻结**：所有评测脚本吃版本号，不吃"最新版"——保证跨时点分数可比

详细方案见 `PROJECT.md`，每日实施轨迹见 `PLAN.md`。

# 二、课题要求
## 课题名称
“EvalForge：评测集炼成工坊”——面向AI Agent的评测数据自动化构建与版本化更新维护
## 课题要求
当前大模型与AI Agent的能力评测高度依赖评测集（Benchmark）。然而：
- 静态评测集易过时：模型不断演进，原有评测题被“刷爆”
- 人工构建评测集成本高：单条高质量用例需数分钟
- 缺乏自动化更新机制：无法快速响应新风险/新任务
本课题旨在构建一个半自动化的评测集生产线，聚焦两大核心环节：
1. 评测集构建：利用大模型生成、人工校验、质量过滤的多阶段流水线
2. 评测集更新维护：实现增量更新、版本追溯、衰退检测与自动扩充
## 任务拆解
### 工程架构


| 阶段 | 关键任务 | 交付物 |
| :--- | :--- | :--- |
| 1. 需求分析与规格定义 | · 选定垂直领域（文档编写/知识检索/数据分析/skill生成任选其一）<br>· 定义评测任务类型与输入输出规格<br>· 确定评测集应覆盖的难度梯度、场景分布（正常/边界/对抗） | 领域需求文档、任务规格说明书、场景覆盖矩阵 |
| 2. 种子库构建与评测集生成 | · 人工编写30-50条高质量种子用例<br>· 设计LLM生成模板（自问自答、链式思考、对抗变异）<br>· 实现去重与多样性过滤器 | 种子用例库、数据生成脚本 |
| 3. 自动化质量验证与清洗 | · 开发规则校验器（格式、答案可判定性、无歧义性）<br>· 引入LLM-as-Judge进行语义一致性与难度评分<br>· 执行自动化清洗，标记低质量用例 | 质检模块、质量报告 |
| 4. 人工校验与版本基线发布 | · 交叉人工复核抽样用例（建议20%抽检）<br>· 修正错误标注或模糊问题 | 正式评测集v1.0（≥500条） |
| 5. 增量更新与失效检测机制 | · 实现"失效用例检测器"：基于目标模型在评测集上的表现，识别准确率>90%的“过时用例”<br>· 设计增量替换策略：针对失效用例，调用LLM重新生成同语义不同难度的新用例 | 增量更新能力、版本管理能力、更新日志 |
| 6. 评测集应用与反馈闭环 | · 使用最终评测集对至少2个公开Agent（如DeepSeek、GPT）进行实测打分<br>· 收集Agent评测的Badcase，反向注入生成器作为新种子<br>· 输出评测报告与评测集改进建议 | Agent评测报告、Badcase分析文档、下一版本更新计划 |


### 加分项
- 动态评测集：实现基于在线模型反馈的自动扩充（例如接入开放API，收集失败case反向注入数据集）
- 大模型作为评判器：使用LLM自动评估新增用例的质量并给出建议
- 多模态支持：评测集中加入图像/表格等多模态输入（若选择多模态领域）
- 跨领域泛化：生成的评测集可迁移到至少两个不同任务（如代码 + 安全）
### FAQ
Q1：评测集更新维护的“维护”具体指什么？
A：两个维度：① 版本迭代——当旧评测集被模型刷高分后，自动替换失效用例；② 质量维护——定期运行衰退检测，若发现评测集区分度下降（所有模型准确率>90%），触发重构。
Q2：评测集中的“真值”（Ground Truth）如何确保绝对正确？如果任务本身是开放性的（例如文案生成、代码风格优化），怎么定义标准答案？
A：这是评测集构建中最关键的挑战。可采用分层策略应对不同任务类型：



| 任务类型 | 真值获取策略 | 示例 |
| :--- | :--- | :--- |
| 确定性任务（代码运行、数学计算、SQL查询） | 实际执行验证：通过沙箱环境运行代码/查询，对比输出结果。真值=预期输出。 | “编写函数计算斐波那契数列第n项”→ 运行 Python 代码验证。 |
| 半确定性任务（分类、信息抽取、多项选择） | 规则判定 + 多模型交叉标注。若两个大模型答案不一致，标记为“争议用例”需人工介入。 | “判断以下评论的情感极性”→ 采用多数投票或人工裁定。 |
| 开放性任务（文案生成、代码重构、创意建议） | 使用 LLM-as-Judge + 维度分解评分：不要求唯一答案，而是定义若干可量化维度（如流畅度、信息量、安全性），用大模型打分并设置一致性阈值。 | “写一段吸引人的产品介绍”→ 评分维度：吸引力（1-5）、信息准确度（1-5）、无违规（0/1）。 |


Q3：有没有可以参考的公开评测集？
当然可以。参考现有优秀评测集的设计思路甚至复用部分数据，是高效构建自己评测集的常见做法。
