## 二、为什么要做"质检"——上次答疑被忽略的一个问题

上次结尾我讲了 591 条候选，老师有同学问过：**"质量怎么保证？"**

我当时只能答"种子写得严"——但这其实远远不够。因为：

1. **参数膨胀** 会复制错误：种子里如果有一个 USR_99（少一位）的笔误，膨胀后 20 条全错。
2. **LLM 改写** 可能引入新问题：改写时如果把"DELETE 权限"说成"删除权限"，看着没问题，但 prompt 跟 expected 的语义距离就拉大了，影响可判定性。
3. **自动去重** 只去重复，不去烂题。

所以我必须**主动找出烂题、淘汰掉**。这就是阶段 3+4 的全部目的。

## 三、阶段 3：自动质检（D5）

> **核心思路：分两层把关——程序保下限，LLM 保上限。**

### 1. 第一层：JSON Schema 强校验（程序，秒级）

对每条候选做这五件事：
1. 必需字段齐全（task_id / prompt / expected_tool_calls / difficulty / skill_name）
2. skill_name 必须在 4 个 Skill 之内
3. difficulty 必须是 normal / boundary / adversarial 之一
4. 每个工具调用，**工具名必须在 9 个原子工具内**、**参数必须符合 JSON Schema**
5. 工具必须在该 Skill 的 `allowed_tools` 白名单内（比如 SecureAdmin 不能调 Mask_PII）

**结果：591 条全部通过，零淘汰。**

这一层有点像"语法检查"——本来就该全过，没过的是结构性 bug，必须修。**没出问题恰恰说明前面种子+膨胀+改写这条链路是干净的**。

### 2. 第二层：LLM-as-Judge 三维打分（DeepSeek）

光通过 schema 不够——题可能"语法对、语义烂"。比如：
- prompt 太短，没有可判定的标准答案（可判定性差）
- 标着 normal 但其实是 boundary（难度标签错）
- 中文别扭，机翻味重（流畅度差）

所以我用 DeepSeek 给每条题打三个维度的分（1-5 整数）：

| 维度 | 定义 | 为什么是这个 |
|---|---|---|
| **可判定性** decidability | 程序能不能算分，标准答案唯一吗 | 这是我整套 benchmark 的存在前提 |
| **难度合理性** difficulty_fit | 题的实际难度跟标签贴不贴 | 难度标签是后面演化的基础 |
| **流畅度** fluency | 中文通顺、自然 | 不通顺会让 Agent 把"不会"误判成"读不懂" |

**为什么不直接判分？为什么打三维？**
- 单一总分掩盖问题：4 分平均分可能是"流畅 5 分、可判定 3 分"，也可能是"流畅 5 分、难度合理 3 分"，对应的修法完全不同
- 三维独立打 → 哪一维出问题，能定位哪个环节坏

**淘汰阈值**：三维任一 < 3 分 → 淘汰。

### 3. 真实跑出来的数据

跑完一轮 DeepSeek（97.3 秒、约 ¥0.2、零 fallback）：

| 维度 | 均分 | <3 数量（即被淘汰）|
|---|---|---|
| 可判定性 | **4.89** | 0 |
| 难度合理性 | **4.92** | 0 |
| 流畅度 | **4.98** | 0 |

- 591 条全部保留 → v0.95
- 单维度最低 3 分的只有 5 条（仍保留）

### 4. 我要坦诚一个问题

老师肯定会问：**"全过，是不是阈值太松？"**

我承认这是个值得追问的点。我自己也想过两个解释：
1. **样本基线本来就高** —— v0.9 是 40 条手写种子膨胀来的，种子是我亲自写的最高优先级保证；膨胀和改写都是受控变化，没引入新错误源。所以"几乎全过"是数据自身决定的，不是阈值松。
2. **但单跑 LLM 不能完全证明这点** —— 这正是为什么必须再加阶段 4 的人工抽检，让人脑跟 LLM 交叉验证。

下面就讲人工抽检。

## 四、阶段 4：人工抽检 + v1.0 发布（D6）

### 1. 为什么自动质检之后还要人工

> **LLM-as-Judge 有一个天然漏洞——它自己也会错。**

如果 v0.95 完全相信 LLM Judge 的打分，那项目就变成"用 LLM 评 LLM"——评价标准跟被评测对象同源，循环论证。

所以必须有一层**人脑**当兜底。**人不需要全审，只需要抽样审**——核心目标是验证 LLM Judge 自己有没有大面积放水。

### 2. 怎么抽——分层随机 20%

- **不分层随机抽**会出问题：591 条里 adversarial 才 60 条（10%），全部随机抽 20% 可能就抽中 1-2 条对抗题，根本验不了对抗题质量。
- 所以按 **difficulty 分层抽样**：
  - normal 353 × 20% = 71 条
  - boundary 178 × 20% = 36 条
  - adversarial 60 × 20% = 12 条
  - **合计 119 条**

20% 是参考工业界经验值——质检界常用的 ANSI 抽样标准也大致在 10-20% 区间。

### 3. CLI 工具长什么样

我写了个命令行小工具，跑起来一条一条弹给你，键盘做决策：

```
[1/119]  T_CTH_001_V01
  Skill      : CustomerTicketHandling
  Difficulty : normal
  Prompt:
    用户 USR_001 申请退货 ITEM_9999，已确认收货，无法退货，需要建议人工售后……
  Expected tool calls:
    1. Fetch_User_Data({"user_id": "USR_001"})
    2. Check_Inventory({"item_id": "ITEM_9999"})
    3. Create_Ticket(...)
    4. Send_Notification(...)
  键盘：[y]通过  [n]淘汰  [m]修改  [s]跳过  [q]退出
```

**4 个键 + 2 个工程考虑**：
- `y / n / m / s` 四种决策，`m`（修改）会让我留一行 note
- 每 5 条自动落盘 → 断电不丢
- 重启脚本会自动跳过已审过的 → 中断恢复

### 5. 发布 v1.0

抽检结束后，跑 `python -m release`：

1. 应用决策：`reject` 剔除、`modify` 保留但打 `needs_modify` tag
2. 再跑一次 schema 强校验（双保险）
3. 写入 `data/version_history/benchmark_v1.0.0.json`（**冻结副本，永不修改**）
4. 自动更新 `CHANGELOG.md`，记录本次发布的总数 + 决策分布

为什么版本冻结很重要：以后所有的评测分数都要对得上某一个**确定版本号**。否则今天跑的 88 分和下周跑的 91 分根本没法比——可能不是模型变好，是数据集偷偷变了。

## 附录：预想可能被问到的问题

**Q：抽检 20% 够吗？为什么不是全审？**
答：120 条已经是单人 2 小时的工作量。工业界质检通用经验值是 10-20%（ANSI Z1.4 抽样标准）。再加上抽样是**按 difficulty 分层**的——每一层都被检验到了，不是简单 20% 平均。如果时间允许，框架支持随时再加抽，只需重跑 CLI，记录会自动叠加。

**Q：LLM-as-Judge 自己也会错，那它给的分数还有意义吗？**
答：LLM Judge 的作用是**批量过粗筛**——它能把 90%+ 没问题的题快速放过，让人工 20% 抽检集中精力。整套机制是 **"程序保下限 + LLM 保中段 + 人工保上限"** 三层。LLM 错了，人工会兜住——这就是为什么要有 20% 抽检。这次抽检我实际 reject 了【】条 LLM 漏过的题，就是证据。

**Q：三个维度的均分都是 4.89-4.98 接近满分，是不是 LLM 给打分太松？**
答：我自己也想过这个问题。有两个可能性：
- (a) 评分太松 →  人工抽检应该能拒掉 5-10% 才合理
- (b) 数据基线本来就高 →  人工抽检大部分会同意
我的抽检实测结果是【】，倾向解释【】。即使是 (a)，prompt 里 1-5 分的描述足够清晰，再严的 LLM 也不太可能压到 3 以下；要更严格的话可以下次答疑前换成 7 分制重跑。

**Q：为什么 cutoff 是 < 3 不是 < 4？**
答：3 分是"勉强可用"的下限，<3 才是"有明显问题"。如果设 <4，会把大量"挺好只是不算惊艳"的题一起淘汰，损失 30% 数据量，**得不偿失**。这种边界值我留了 cutoff 参数，可以一行代码调整。

**Q：你 reject 的题，理由能复现吗？会不会换个人就 accept 了？**
答：我的判定标准是**"如果两个工程师独立读这条题，必须能写出同一个 expected"**。reject 的题至少违反这一条。这个标准是可复现的——下次找另一个同学按同样标准复抽一遍，结果会高度一致。这是我用"硬规矩"代替"个人偏好"的设计选择。

**Q：v1.0 发布后，candidates_v0.95.json 还会动吗？**
答：不会。**v1.0 是冻结快照**，存在 `data/version_history/benchmark_v1.0.0.json`。后面 v1.1（衰退演化版本）也是另一个冻结文件，永远可对照。这是为什么所有评测脚本都吃版本号，而不吃"最新版"。

**Q：CHANGELOG 有什么用？**
答：每发布一个版本，CHANGELOG 自动追加一行：总数、决策分布、日期。等到 6/2 我有 v1.1 之后，CHANGELOG 就成了**版本演化证据链**——可以直接看出每个版本相对前一个改了什么。

**Q：人工抽检的 log 还能用来干嘛？**
答：除了支撑 v1.0 发布之外，**modify 标签的 note 是后续优化的种子**——后面 D10 反向注入阶段，我会把 note 转化成新的种子模式回流。这一步把"人工劳动"变成了"种子库的反向增量"。

**Q：你设计了 4 种决策键 y/n/m/s，但其实你只用了 y/n/m，为什么留 s？**
答：`s` 是"我看不准，先跳过"——人审 119 条会累，到第 80 条可能脑子糊。s 是给疲劳保留的容错。这次抽检里 skip 占【】%。skip 的题之后可以二次回审，不浪费。
